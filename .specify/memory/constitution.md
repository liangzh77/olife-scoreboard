# OLife 项目宪章

## 核心原则

### I. 输入文档保护

`configuration` 文件夹和 `api` 文件夹是项目的输入文档，包含产品需求和API规范。
这些文件夹内的内容为只读，任何情况下都不得修改。

- 必须将 `configuration/` 视为只读的产品需求文档
- 必须将 `api/` 视为只读的API规范文档
- 如需基于输入文档创建派生文档，必须在 `specs/` 文件夹中维护

**理由**: 保持输入文档的完整性，确保需求源头不被意外修改，便于追溯和版本管理。

### II. 文档规范

所有项目文档和用户沟通必须使用中文。

- 文档内容必须使用中文撰写
- 代码注释推荐使用中文（技术术语除外）
- 与用户的所有沟通必须使用中文
- 自有文档必须存放在 `specs/` 文件夹中

**理由**: 确保团队成员能够准确理解项目文档，减少因语言障碍造成的误解。

### III. uni-app 开发规范

`track-screen` 是基于 uni-app 框架的主工程目录。

- 必须遵循 uni-app 官方开发规范和最佳实践
- 组件开发必须考虑多端兼容性（微信小程序、H5、App等）
- 页面路由必须在 `pages.json` 中正确配置
- 样式编写推荐使用 rpx 单位以确保多端适配

**理由**: uni-app 的核心价值在于"一次开发，多端发布"，遵循规范可最大化跨平台复用。

### IV. 跨平台兼容性

开发功能时必须考虑目标平台的差异。

- 必须在使用平台特定API前进行条件编译或能力检测
- 必须使用 `#ifdef` / `#ifndef` 处理平台差异代码
- 避免使用仅在特定平台可用的原生API，除非有明确的降级方案

**理由**: 确保应用在所有目标平台上都能正常运行，避免运行时错误。

### V. 简洁性原则

保持代码和架构的简洁性，避免过度工程化。

- 只实现当前明确需要的功能，遵循 YAGNI 原则
- 优先使用 uni-app 内置组件和API
- 避免引入不必要的第三方依赖
- 代码应当自文档化，复杂逻辑需添加必要注释

**理由**: 简洁的代码更易维护、更易测试、更少bug。

## 技术约束

### 技术栈

- **框架**: uni-app (Vue.js)
- **开发语言**: JavaScript / TypeScript
- **样式方案**: SCSS/CSS，使用 rpx 单位
- **目标平台**: 根据项目需求确定（微信小程序、H5、App等）

### 项目结构

```text
olife/
├── configuration/     # 只读 - 产品需求文档
├── api/              # 只读 - API规范文档
├── track-screen/     # uni-app 主工程
│   ├── pages/        # 页面目录
│   ├── components/   # 组件目录
│   ├── static/       # 静态资源
│   └── ...
└── specs/            # 自有文档目录
```

## 开发工作流

### 代码审查要求

- 所有代码变更必须验证是否符合本宪章规定的原则
- 必须验证未修改 `configuration/` 和 `api/` 目录下的任何文件
- 必须验证跨平台兼容性

### 文档更新

- 功能变更必须同步更新 `specs/` 中的相关文档
- 文档必须保持与代码实现的一致性

## 治理

本宪章是项目开发的最高指导原则，优先于其他所有开发实践。

### 修订程序

1. 提出修订建议并记录修订理由
2. 团队评审讨论
3. 更新本文档并记录版本变更
4. 同步更新所有依赖本宪章的模板和文档

### 版本策略

- **MAJOR**: 原则的删除或重新定义（不向后兼容）
- **MINOR**: 新增原则或重要指导的扩展
- **PATCH**: 澄清、措辞修正、非语义性改进

### 合规检查

- 所有 PR 和代码审查必须验证是否符合本宪章
- 使用 CLAUDE.md 作为运行时开发指导参考
